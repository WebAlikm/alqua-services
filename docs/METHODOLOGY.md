# Methodology

This project uses an IB Computer Science IA-inspired workflow. The goal is not to copy the school format, but to use its strongest habits: define the problem, justify decisions, set measurable success criteria, build iteratively, and evaluate with evidence.

## Why This Method Fits the Hackathon

Tatweer Hackathon round one is judged entirely from the GitHub repository. A structured methodology helps judges see that the project is not only a prototype, but a planned solution with clear evidence and realistic deployment assumptions.

## Method Overview

| Stage | Output | Repo Evidence |
| --- | --- | --- |
| Investigation | Community context and user needs | `docs/community-insights.md`, `docs/rationale.md` |
| Planning | Scope, success criteria, and schedule | `docs/success-criteria.md`, `docs/timeline.md` |
| Design | Architecture and technical decisions | `docs/architecture.md`, `docs/technical-justification.md` |
| Development | Working resident discovery prototype | `src/`, `data/`, `index.html` |
| Testing | Manual checks and performance notes | `evidence/user-testing.md`, `evidence/performance.md` |
| Evaluation | Criteria-based scoring and claims | `docs/scoring-tracker.md`, `docs/criteria/`, `evidence/README.md` |

## Core Research Question

How can a mobile-first, Arabic-first web app help residents in a dispersed rural community discover relevant local services, opportunities, and events faster than fragmented informal channels?

## Development Principles

- Build the smallest complete end-to-end resident flow first.
- Prioritize mobile usability over complex features.
- Make all impact claims testable.
- Label proxy data clearly when village-level data is unavailable.
- Keep deployment realistic for a rural community setting.
- Store evidence in the repository, not only in the final presentation.

## Evaluation Method

The prototype will be evaluated against:

1. Hackathon judging criteria.
2. Project-specific success criteria.
3. Manual user-flow tests.
4. Evidence from sample listings, screenshots, and demo video.
5. Feasibility and scalability documentation.

## Methodology Footnote

Data synthesized using Abu Dhabi Census reports, TDRA digital government and telecom context, UAE digital adoption metrics, Tatweer Hackathon challenge material, and regional community news coverage of the Al Ain desert region. Village-level Al Qua'a demographic figures were not found in public official sources, so this repo labels regional or national proxy data clearly.

## Source Notes

- SCAD is used for Abu Dhabi and Al Ain regional census context.
- DataReportal is used for UAE national digital adoption context.
- TDRA is used as official UAE telecom and digital government context, not as village-level evidence.
- Tatweer Hackathon materials are used as the authoritative source for the challenge problem statement.
