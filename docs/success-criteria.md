# Success Criteria

These criteria are defined before implementation. They are intentionally few, measurable, and easy for judges to verify.

| ID | Criterion | Target | Evidence |
| --- | --- | --- | --- |
| SC1 | Fast access | Page loads in under 3 seconds on a throttled mobile connection | `evidence/performance.md` |
| SC2 | Useful coverage | At least 20 listings across 3 or more categories | `evidence/service-coverage.md` |
| SC3 | Quick discovery | A resident finds a relevant listing in 3 clicks or fewer | `evidence/user-testing.md` |
| SC4 | Arabic usability | Arabic layout is readable and does not break on mobile | `evidence/screenshots/` |
| SC5 | Accessible interface | No critical accessibility issues in automated checks | `evidence/accessibility.md` |
| SC6 | Reusable model | Services, events, and opportunities use one listing structure | `data/sample-listings.json`, `evidence/service-coverage.md` |

## Completion Definition

The project is ready for submission when:

- The app runs locally.
- The README includes run steps and demo status.
- Each success criterion has evidence or a clearly marked limitation.
- Screenshots or a demo video prove the main user flow.
- The repo does not contain unsupported claims.
